# create-issue スキル

作業中に発見した課題を、重複を避けつつ本リポジトリの開発 Issue として起票するスキルです。

## 呼び出し方

```
/create-issue <課題の要約 or 根拠 or 関連 URL>
```

例: `/create-issue setup-labels.sh で Organization 対応が不足している`

引数は `$ARGUMENTS` として参照します。

## 実行手順

### ステップ1: スコープの確定

1. `$ARGUMENTS`・現在の作業内容・関連する親 Issue / PR を読み、課題を理解する
2. 独立した Issue として追跡すべき課題かを判断する
3. 事実と未確認事項を切り分ける（推測を事実として書かない）

### ステップ2: 重複の確認

4. Open / Closed 両方の Issue を横断して重複を確認する:
   ```bash
   gh issue list --state all --search "<キーワード>" --limit 100
   gh issue view <番号>
   ```
   関連しそうなキーワードで複数回検索する。重複が見つかった場合は**起票を中止**し、既存 Issue の URL と理由を報告する。

### ステップ3: ラベルの選定

5. `gh label list` で利用可能なラベルを確認する
6. 課題の性質に最も近い**既存ラベル**を1つ以上選ぶ:
   - バグ・不具合 → `bug`
   - 機能追加・改善・リファクタ → `enhancement`
   - ドキュメント → `documentation`
   - 疑問・要調査 → `question`
7. 適切なラベルが存在しない場合は、新規作成せず既存ラベルから最も近いものを選ぶ

### ステップ4: 本文の作成

8. 課題を追跡可能な形で本文にまとめる。以下の見出し構成で記述する:
   - **背景 / 目的**: なぜこの Issue が必要か
   - **現状 / 事実**: 確認済みの事実（ファイルパス・行番号・再現手順など）
   - **未確認事項**: 不明点は `未確認`、該当なしは `なし` と明記する
   - **完了条件**: 検証可能なチェックボックス（`- [ ] ...`）で記述する
   - **関連情報**: 親 Issue / PR / ドキュメントの URL
9. 本文は一時ファイルに書き出し、`--body-file` で渡す（scratchpad を利用）

### ステップ5: セルフレビュー

10. 起票前に以下を確認する:
    - タイトルが「問題」と「期待する変化」を簡潔に表しているか
    - 事実と未確認事項が分離されているか
    - スコープ・除外範囲・完了条件が明示されているか
    - 親 Issue / PR へのリンクがあるか
    - 選んだラベルが実在し、課題に合致するか

### ステップ6: 曖昧点の確認（必要時のみ）

11. 正確な Issue 化を妨げる**重大な曖昧さ**がある場合のみユーザーに確認する。軽微な不明点は `未確認` として本文に残す。

### ステップ7: 起票と確認

12. Issue を作成し、内容を検証する:
    ```bash
    gh issue create --title "<タイトル>" --body-file "<本文ファイル>" --label "<label>"
    gh issue view "<issue-url>" --json url,title,body,labels
    ```

### ステップ8: 完了報告

13. 以下をユーザーに報告する:
    - 起票した Issue の URL
    - 要約（1〜2文）
    - 付与したラベル
    - 関連づけた親 Issue / PR
